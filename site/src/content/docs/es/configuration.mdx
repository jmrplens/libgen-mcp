`libgen-mcp` se configura por completo mediante variables de entorno. Todas las variables son opcionales: un valor vacío o sin definir usa el valor por defecto documentado.

<Aside type="caution">
	Una variable presente pero mal formada (un número incorrecto, un valor fuera
	de rango, un directorio no escribible, un nombre de fuente desconocido) hace
	que el servidor **falle de inmediato al arrancar** con un error explicativo,
	en lugar de recurrir silenciosamente al valor por defecto.
</Aside>

Fija estas variables en el bloque `env` de tu cliente MCP (consulta [Primeros pasos](/libgen-mcp/es/getting-started/)), en tu shell, o con flags `-e` en `docker run`.

## Notas sobre variables concretas

### `LIBGEN_MIRROR`

Cuando se define, este mirror se usa como el único mirror preferido y se coloca primero. Debe parsearse como una URL `http`/`https` con host, de lo contrario el arranque falla. Déjala sin definir para que el servidor descubra los mirrors vivos actuales y haga failover entre ellos automáticamente.

### `LIBGEN_MCP_DOWNLOAD_DIR`

Al arrancar, el directorio se crea (con modo `0750`) si no existe y se escribe y elimina un archivo temporal para confirmar que es escribible. Bajo Docker, el contenedor se ejecuta como UID `10001`, así que un directorio de host montado debe ser escribible por ese usuario.

### `LIBGEN_MCP_UNPAYWALL_EMAIL`

Solo la usa la fuente `unpaywall` (DOIs de artículos). Está **sin definir por defecto**, lo que **desactiva la fuente `unpaywall`**: la API de Unpaywall exige un email de contacto en cada consulta y rechaza las peticiones sin él. Fija tu propia dirección de contacto para activarla. Un valor vacío es válido (desactivado); un valor no vacío se comprueba superficialmente (debe contener `@` y un dominio con punto) pero no se verifica su entregabilidad.

### `LIBGEN_MCP_DOWNLOAD_START_RETRY_WAITS` y `LIBGEN_MCP_DOWNLOAD_STALL_TIMEOUT`

Estas dos variables hacen que las descargas sean resilientes sin cortar nunca una transferencia sana pero lenta:

- **Arrancar** se reintenta con un programa escalonado. Una descarga primero debe _iniciarse_ — resolver una URL nueva, conectar y entregar su primer byte. Si algo de eso falla (error de resolución, error de conexión, estado no 2xx, o un 2xx que no entrega bytes), el intento se reintenta tras la siguiente espera de `LIBGEN_MCP_DOWNLOAD_START_RETRY_WAITS`. El valor por defecto `5s,5s,5s,10s,10s,10s,15s` realiza 8 intentos en unos 60 segundos; cada reintento resuelve de nuevo, de modo que una clave caducada se renueva. Esto envuelve a una sola fuente; la cadena multifuente sigue avanzando a la siguiente fuente cuando una se agota. Cuando todas las fuentes fallan al iniciar, la herramienta devuelve un error accionable que indica al modelo que puede reintentar ahora, reintentar más tarde o preguntar al usuario.
- **Seguir viva** lo rige `LIBGEN_MCP_DOWNLOAD_STALL_TIMEOUT`, una ventana que _se reinicia con el progreso_. Una vez que fluyen bytes, la transferencia se aborta solo si **no** llega ningún byte nuevo dentro de la ventana; cada byte recibido reinicia el reloj. Una descarga a 30 kB/s que tarda muchos minutos nunca se corta, mientras que una conexión realmente estancada se descarta tras la ventana y su `.part` se conserva para reanudar después. El `LIBGEN_MCP_TIMEOUT` por petición nunca se aplica a una descarga en streaming.

### `LIBGEN_MCP_SOURCES` y la cadena de fuentes

La cadena de descarga siempre se ejecuta en el orden fijo `unpaywall → scihub → scidb → libgen → randombook → annas`. Cada fuente declara qué elementos admite, de modo que en la práctica:

- **Los libros** (identificados por `md5`) se prueban contra `libgen`, luego `randombook`, luego `annas`.
- **Los artículos** (identificados por `doi`) se prueban contra `unpaywall` (solo cuando `LIBGEN_MCP_UNPAYWALL_EMAIL` está fijado), luego `scihub`, luego `scidb`.

`scidb` sirve artículos académicos a través del visor SciDB de Anna's Archive. No necesita clave y se sitúa después de `scihub` porque alcanza artículos publicados tras el cese de indexación de Sci-Hub, cubriendo ese hueco en lugar de sustituirlo.

`annas` es el último rescate para libros y por defecto no necesita clave: las páginas de libro de Anna's publican el CID de IPFS de cada elemento, y la fuente transmite desde el primer gateway público de IPFS que lo sirva. La disponibilidad de los gateways varía mucho, así que conviene tratarlo como un fallback real y no como una vía rápida: para elementos poco sembrados puede ser lento o agotar el tiempo. Fijar `LIBGEN_MCP_ANNAS_KEY` hace que se pruebe primero la API de miembro, mucho más rápida, con IPFS todavía como fallback.

`LIBGEN_MCP_SOURCES` solo elimina fuentes de esa cadena; nunca la reordena. Por ejemplo, `LIBGEN_MCP_SOURCES=libgen` desactiva las fuentes de artículos y el fallback `randombook`, dejando solo la fuente principal de libros de LibGen. Consulta [Arquitectura](/libgen-mcp/es/architecture/) para saber cómo se construye y recorre la cadena.

### `LIBGEN_MCP_REMOTE_DOWNLOADS`

Pone al servidor en el mismo **modo de descarga remota** que activa automáticamente `--http`, pero para el transporte **stdio**. Esto importa en un despliegue stdio alojado — por ejemplo el servidor ejecutándose detrás de `mcp-proxy` en un catálogo como Glama — donde el proceso corre en una máquina remota o efímera cuyo disco el cliente no puede alcanzar, así que guardar un archivo ahí no sirve de nada. Cuando se fija a un valor verdadero, cada llamada a `download` devuelve siempre un enlace (un bloque de contenido `resource_link` más un objeto `resolved`, con las `headers` necesarias) en lugar de escribir en disco, igual que ya hace bajo `--http`. Acepta `1`/`true`/`0`/`false`; cualquier otro valor hace fallar el arranque. Consulta [Herramientas](/libgen-mcp/es/tools/#dónde-acaba-el-archivo-local-vs-remoto).

### `LIBGEN_MCP_READ_MAX_CHARS` y `LIBGEN_MCP_READ_DEFAULT_PAGES`

Estas dos variables acotan cuánto texto devuelve por defecto una sola llamada a `read`: `LIBGEN_MCP_READ_MAX_CHARS` limita los fragmentos de EPUB/TXT por número de caracteres, `LIBGEN_MCP_READ_DEFAULT_PAGES` limita los fragmentos de PDF por número de páginas. Ambas son solo valores por defecto — el propio argumento `max_chars`/`max_pages` de una llamada siempre tiene prioridad cuando se fija a un valor positivo. Súbelas para devolver más texto por llamada (menos idas y vueltas, más tokens por respuesta) o bájalas para mantener las respuestas individuales pequeñas. Consulta [Herramientas](/libgen-mcp/es/tools/#read) para la referencia completa de `read`.

### `LIBGEN_MCP_READ_CACHE_BYTES` y `LIBGEN_MCP_READ_CACHE_TTL`

`read` obtiene un archivo identificado por `md5`/`doi` a un archivo temporal en el servidor una sola vez, y luego sirve las páginas/fragmentos sucesivos de ese mismo archivo mientras el modelo pagina con `cursor` — sin esta caché, cada página volvería a descargar el archivo completo. `LIBGEN_MCP_READ_CACHE_TTL` es cuánto tiempo permanece un archivo temporal sin referenciar tras su último uso antes de desalojarse; `LIBGEN_MCP_READ_CACHE_BYTES` es el techo de tamaño total de la caché, por encima del cual se desalojan primero los archivos usados menos recientemente (un archivo que una llamada a `read` está usando activamente nunca se desaloja). Ninguna de las dos variables afecta a `download`, que siempre escribe directamente en `LIBGEN_MCP_DOWNLOAD_DIR` y no mantiene caché temporal en el servidor.

### `LIBGEN_MCP_ENRICH`

Controla si el argumento opt-in `enrich: true` de `get_details` se respeta en absoluto. El valor por defecto `true` significa que una llamada _puede_ pedir enriquecimiento (sigue desactivado a menos que una llamada fije `enrich: true`); fijarlo a `false` hace que el servidor ignore `enrich` por completo, así que nunca se hacen llamadas a Crossref/OpenLibrary en este despliegue. Úsalo para desactivar por completo las llamadas salientes a esas dos APIs de terceros, por ejemplo en un despliegue restringido o que prefiere funcionar sin conexión. Consulta [Herramientas](/libgen-mcp/es/tools/#enriquecimiento-de-metadatos) para ver qué añade el enriquecimiento y su presupuesto best-effort de 6 segundos.

### `LIBGEN_MCP_OPEN_ACCESS`

Controla el _valor por defecto de despliegue_ para el argumento opt-in `include_open_access` de `search` — no fuerza el descubrimiento a activado o desactivado de forma incondicional, ya que el `include_open_access` propio de una llamada siempre prevalece cuando se fija. El valor por defecto `false` significa que el descubrimiento de acceso abierto (arXiv, Crossref, OpenLibrary) solo se ejecuta en una llamada que fije explícitamente `include_open_access: true`; fijar esta variable a `true` invierte el valor por defecto del despliegue, de modo que el descubrimiento se ejecuta en cada llamada a `search` a menos que una fije explícitamente `include_open_access: false` para excluirse. Los tres proveedores son sin clave (sin cuenta ni API key) y best-effort, cada uno acotado por su propio presupuesto corto por petición, así que un proveedor lento o que falla nunca retrasa ni hace fallar la búsqueda principal de Library Genesis. Consulta [Herramientas](/libgen-mcp/es/tools/#descubrimiento-de-acceso-abierto) para ver la forma de salida fusionada `open_access`.

## Preguntas frecuentes

### ¿Cuál es la configuración mínima de libgen-mcp?

Ninguna. `libgen-mcp` se configura por completo mediante variables de entorno y todas son opcionales: un valor vacío o sin definir usa el valor por defecto documentado, así que el servidor arranca sin ninguna configuración. Eso sí, una variable presente pero mal formada (un número incorrecto, un valor fuera de rango, un directorio no escribible o un nombre de fuente desconocido) hace que el servidor falle de inmediato al arrancar con un error explicativo, en lugar de recurrir silenciosamente al valor por defecto.

### ¿Cómo cambio dónde se guardan las descargas?

Fija `LIBGEN_MCP_DOWNLOAD_DIR` con la ruta del directorio de destino; por defecto es `~/Downloads`. El directorio se crea si falta y se comprueba su escritura al arrancar. También puedes sobreescribir el destino en cada llamada con el argumento `path` de la herramienta `download`. Bajo Docker, monta un volumen escribible por el UID `10001` y apunta la variable a él.

### ¿Cómo descargo artículos por DOI?

Pasa el `doi` de un artículo a la herramienta `download`. Los artículos se obtienen por DOI a través de dos fuentes por orden: primero `unpaywall`, que devuelve un PDF solo cuando el artículo es de acceso abierto, y luego `scihub`, que prueba cada host de `LIBGEN_MCP_SCIHUB_HOSTS` hasta que uno sirve el artículo. La fuente `unpaywall` solo participa cuando fijas tu propia dirección en `LIBGEN_MCP_UNPAYWALL_EMAIL`, que se envía a la API de Unpaywall en cada consulta de artículo; sin ese email `unpaywall` está desactivada y solo se prueba `scihub`. Ten en cuenta que las descargas por DOI no se verifican por MD5.

### ¿Cómo limito la tasa de peticiones?

Ajusta `LIBGEN_MCP_RATE_RPS` (peticiones salientes por segundo permitidas, por defecto `1`, rango `(0, 20]`) y `LIBGEN_MCP_RATE_BURST` (tamaño máximo de ráfaga, por defecto `1`, rango `[1, 100]`). El límite se aplica de forma compartida a todas las peticiones a mirrors y descargas mediante un token-bucket. Baja los valores para ser más suave con los mirrors, o súbelos con cuidado para más rendimiento.
